# Vector VS Code 插件使用说明

## 1. 文档说明

本文档面向 `Vector` 的 VS Code 插件用户，系统介绍插件的定位、安装方式、首次配置、日常使用流程、核心能力、高级功能、常用操作与常见问题。

如果你希望在 VS Code 中使用一个能够理解项目、读写文件、执行终端命令、调用浏览器、接入外部工具，并且在每一步都保持人工审批的 AI 编码助手，`Vector` 就是为这类工作流设计的。

## 2. Vector 是什么

`Vector` 是一款运行在 VS Code 内部的智能编码代理。与传统代码补全工具不同，`Vector` 不只负责补全单行代码，而是可以围绕一个完整任务持续工作，例如：

- 阅读项目结构并理解代码上下文
- 新建、修改、重构文件
- 运行测试、构建、脚本和常用开发命令
- 启动浏览器并执行页面交互或调试操作
- 结合 MCP 扩展外部工具能力
- 在复杂任务中先规划、再执行

`Vector` 的核心特点不是“自动执行一切”，而是“在你的控制下完成复杂开发任务”。无论是文件改动、终端命令还是浏览器动作，都会先展示给你，再由你决定是否批准执行。

## 3. 适用场景

`Vector` 特别适合以下场景：

- 在已有项目中快速定位问题、理解模块关系并实施修改
- 根据自然语言需求完成一个功能点的设计与实现
- 修复编译错误、测试失败、运行时报错或页面交互问题
- 基于终端输出、Git 变更、选中文本和项目文件进行上下文驱动的问题处理
- 需要结合数据库、接口平台、知识库或内部系统能力的复杂研发流程
- 对多文件修改、架构调整、方案评审有较高可控性要求的团队协作环境

## 4. 产品工作方式

`Vector` 的日常工作模式可以概括为以下流程：

1. 你描述目标、问题或需求。
2. `Vector` 读取相关上下文，例如文件、目录、问题列表、终端输出或网页内容。
3. `Vector` 生成计划，必要时向你追问关键约束。
4. 在获得批准后，`Vector` 执行文件编辑、命令运行或浏览器操作。
5. 任务完成后，`Vector` 汇总结果，并保留可追踪的改动历史。

这种设计的价值在于：

- 既保留了 AI 的自动化能力，也保留了开发者对代码库的最终控制权
- 更适合真实项目，不依赖“把所有背景一次性复制进对话框”
- 对复杂任务的支持明显强于单轮问答型工具

## 5. 安装与启动

### 5.1 安装插件

在 VS Code 中打开扩展市场：

- Windows / Linux：`Ctrl+Shift+X`
- macOS：`Cmd+Shift+X`

搜索 `Vector`，点击安装。

### 5.2 打开插件入口

安装完成后，VS Code 左侧 `Activity Bar` 会出现 `Vector` 图标。点击后即可打开 `Vector` 侧边栏面板。

插件的主入口通常包括以下操作：

- `New Task`：新建任务
- `MCP Servers`：查看和管理 MCP 服务
- `History`：查看历史任务
- `Account`：查看账号相关信息
- `Settings`：进入插件设置

## 6. 首次使用前的准备

首次打开 `Vector` 后，通常会看到欢迎界面。你有两种接入模型的方式：

- 使用 `Vector` 账户登录
- 使用你自己的 API Key 或模型服务

### 6.1 方式一：使用 Vector 账户

这是最适合大多数用户的入门方式。你只需要在侧边栏中点击开始或登录按钮，完成账户认证后，就可以直接使用平台提供的模型能力。

这类方式适合：

- 希望尽快开始使用，不想自行管理多个模型平台密钥
- 希望统一使用计费与模型入口
- 希望更快完成首次配置

### 6.2 方式二：使用自有模型或 API Key

如果你已经拥有模型服务账号，或者你需要更细粒度的模型控制，可以在设置中切换为自定义 Provider，并填写相应的密钥或连接信息。

这类方式适合：

- 企业或团队已有固定模型供应商
- 需要按自有账户计费
- 需要本地模型、私有部署或定制化连接方式

### 6.3 在哪里配置模型

`Vector` 的主要模型配置不依赖 VS Code 原生 Settings 页面，而是在 `Vector` 面板内的 `Settings` 页面中完成。

设置页的主要分区包括：

- `API Configuration`
- `Features`
- `Browser`
- `Terminal`
- `General`
- `About`

部分组织环境下还可能出现 `Remote Config`。

## 7. 模型与 Provider 配置

### 7.1 API Configuration

在 `Settings > API Configuration` 中，你可以完成：

- 选择 AI Provider
- 配置 API Key 或登录状态
- 选择当前使用的模型
- 设置 `Plan` 与 `Act` 是否使用不同模型

### 7.2 建议的配置思路

如果你不确定如何选，建议按下面的思路配置：

- 偏重推理和方案设计：为 `Plan Mode` 选择更强的推理模型
- 偏重实现速度和成本控制：为 `Act Mode` 选择更快或更经济的模型
- 以隐私和本地化为优先：使用本地模型方案
- 希望快速上手：优先使用默认登录路径

### 7.3 Plan / Act 分离模型

`Vector` 支持为 `Plan Mode` 和 `Act Mode` 配置不同模型。启用后：

- 你在规划阶段可以使用更强的推理模型
- 在执行阶段可以切换到速度更快或成本更低的模型
- 模式切换时，模型与相关 API 配置会按模式分别保留

这对于复杂任务非常有价值，尤其适合“先分析后实施”的开发流程。

## 8. 第一次完成一个任务

下面给出一个推荐的新手上手流程。

### 8.1 新建任务

点击 `New Task`，然后输入一个明确目标，例如：

```text
请为当前项目新增一个字符串工具函数，判断输入是否为回文，并补充对应测试。
```

### 8.2 让 Vector 理解上下文

如果你的任务依赖已有项目结构，建议主动补充上下文，例如：

- 指定相关文件
- 指定相关目录
- 添加终端报错
- 添加工作区问题列表

例如：

```text
请参考 @/src/utils/ 和 @/src/test/ 中已有写法，新增回文判断工具并补充测试。
```

### 8.3 审查 Vector 的行为

接下来，`Vector` 通常会进行以下动作：

- 搜索相关文件
- 阅读已有代码
- 规划实现方案
- 提议修改文件或创建新文件

当它准备落地改动时，会先等待你的批准。你可以：

- 同意执行
- 驳回修改
- 继续补充要求
- 让它先给出计划而不是直接修改

### 8.4 查看结果

任务完成后，建议你重点检查：

- 是否生成了预期文件和代码
- 测试是否覆盖主要路径
- 命名、目录结构、导出方式是否符合当前项目规范
- 是否需要进一步重构或补充说明

## 9. 核心工作模式：Plan 与 Act

`Vector` 的重要特性之一，是将“思考”和“执行”显式区分。

### 9.1 Plan Mode

`Plan Mode` 用于探索、讨论、分析与制定方案。在这个模式下，`Vector` 可以：

- 阅读代码
- 搜索项目
- 理解结构
- 分析设计取舍
- 提出实现建议

但不会直接修改文件或执行实际变更。

适合在以下情况下使用：

- 任务范围不清晰
- 需要先评估架构影响
- 需要排查问题根因
- 想先讨论方案再执行
- 需要做代码评审或风险分析

### 9.2 Act Mode

`Act Mode` 用于落地执行。进入该模式后，`Vector` 才会真正进行：

- 文件创建与编辑
- 终端命令执行
- 浏览器交互
- 自动化实施步骤

### 9.3 推荐实践

对于大多数真实项目任务，建议采用：

1. 先用 `Plan Mode` 让 `Vector` 读懂项目与需求
2. 确认方案合理后切换到 `Act Mode`
3. 在执行过程中继续逐步审批关键动作

这样通常能明显减少返工。

## 10. 上下文添加：让 Vector 更懂你的项目

`Vector` 的表现高度依赖上下文质量。相比一次性贴大量说明，推荐使用精确上下文注入。

### 10.1 常见 `@` 上下文方式

你可以在输入框中通过 `@` 引入上下文，包括：

- `@/路径/文件`：引入单个文件
- `@/路径/目录/`：引入整个目录
- `@problems`：引入当前工作区错误与警告
- `@terminal`：引入最近终端输出
- `@git-changes`：引入未提交变更
- `@提交哈希`：引入指定提交
- `@https://...`：引入网页内容

### 10.2 最推荐的上下文组合

对大多数开发任务，下面几种组合最有效：

- 代码问题定位：`相关文件 + @problems + @terminal`
- 功能开发：`目标目录 + 参考实现文件 + 需求说明`
- 代码审查：`@git-changes + 关键模块文件`
- 文档或接口对齐：`目标代码 + 外部文档 URL`

### 10.3 右键菜单能力

在 VS Code 中，你还可以直接通过右键操作与 `Vector` 协作：

- 编辑器选中文本后可使用 `Add to Vector`
- 代码上下文中可使用 `Explain with Vector`
- 代码上下文中可使用 `Improve with Vector`
- 终端中可使用 `Add to Vector`

这类入口适合快速把“当前正在看的内容”直接带入任务。

## 11. 文件编辑能力

`Vector` 可以直接对工作区文件进行创建、修改与重构，并以可审查的方式向你展示变更。

你可以把它理解为一个会自己理解上下文、提出修改、等待批准、再继续迭代的开发搭档。

### 11.1 适合交给 Vector 的文件类任务

- 新增模块、工具函数、配置文件
- 根据已有模式补充接口、组件、测试
- 统一命名、批量重构、错误修复
- 根据需求整理文档、说明文件、脚本文件

### 11.2 使用建议

- 明确告诉它参考哪些文件和目录
- 提前说明代码风格、目录约定和限制条件
- 让它先解释修改方案，再批准执行
- 对重要改动优先走 `Plan -> Act`

## 12. 终端能力

`Vector` 不仅能写代码，还能在你的开发环境中执行终端命令并读取输出。这使它可以处理更真实的研发流程。

### 12.1 可处理的终端任务

- 安装依赖
- 运行测试
- 启动开发服务
- 执行构建
- 运行脚本
- 根据终端报错继续修复问题

### 12.2 Terminal 设置项

在 `Settings > Terminal` 中，你可以配置：

- 默认终端 Profile
- Shell Integration 超时时间
- 是否启用激进终端复用
- 命令执行模式
- 终端输出行数限制

### 12.3 终端使用建议

- 如果你的环境初始化较慢，可适当提高 Shell Integration 超时时间
- 如果任务频繁卡在终端连接阶段，可先检查终端配置
- 对长时间运行的命令，建议明确告诉 `Vector` 这是开发服务或持续运行进程

## 13. 浏览器能力

`Vector` 可以通过浏览器能力完成前端页面验证、自动化点击、输入、滚动、页面检查等操作。

### 13.1 典型用途

- 测试本地 Web 应用
- 复现并排查前端交互问题
- 检查页面结构、文案、按钮流程和运行时行为
- 结合终端与代码修改形成闭环调试

### 13.2 Browser 设置项

在 `Settings > Browser` 中，你可以配置：

- 是否禁用浏览器工具
- 浏览器视口尺寸
- 是否启用远程浏览器连接
- 远程调试地址
- 自定义 Chrome 路径
- 自定义浏览器启动参数

### 13.3 重要默认行为

浏览器能力默认可能处于关闭状态。如果你希望 `Vector` 直接执行浏览器相关操作，建议先到 `Settings > Browser` 中确认浏览器工具是否已启用。

### 13.4 建议使用方式

适合直接这样描述任务：

```text
启动本地前端并在浏览器中验证登录流程，若页面报错请继续修复。
```

如果你本地已经有运行中的页面，也可以明确指定地址和验证目标，例如：

```text
请打开 http://localhost:3000，检查设置页的保存按钮是否生效。
```

## 14. MCP：为 Vector 接入外部能力

`MCP` 可以理解为 `Vector` 对接外部工具、服务与数据源的标准方式。通过 MCP，`Vector` 可以使用来自外部系统的工具能力，例如：

- 数据库查询
- 项目管理系统操作
- 第三方接口调用
- 内部平台工具
- 文档、知识库、资源读取

### 14.1 MCP 的价值

当任务不再只依赖本地代码，而需要接入外部环境时，MCP 会显著提升 `Vector` 的可用性。

例如：

- 读取工单并据此修复问题
- 查询数据库状态后生成修复方案
- 调用内部 API 完成环境检查
- 结合多个外部工具形成完整自动化流程

### 14.2 如何进入 MCP

你可以通过侧边栏顶部的 `MCP Servers` 进入 MCP 管理界面。

### 14.3 何时使用 MCP

建议在以下场景优先考虑 MCP：

- 本地代码不足以解决问题
- 任务依赖外部系统事实数据
- 需要把 AI 工作流和内部平台打通
- 需要为团队沉淀可复用自动化能力

## 15. 高级功能

`Vector` 不只是单轮对话工具，还提供了一组适合复杂任务的高级能力。

### 15.1 Deep Planning

使用 `/deep-planning` 可以触发更深入的规划流程。它适合：

- 涉及多个文件的大型功能
- 需要架构设计或路径权衡的需求
- 多阶段实施的复杂改造

典型效果包括：

- 系统性探索代码库
- 提出更有针对性的问题
- 生成更完整的实施计划
- 在正式执行前降低方向性错误

### 15.2 Subagents

`Subagents` 允许 `Vector` 把复杂探索任务拆给多个聚焦子代理并行完成。这对于：

- 大型代码库调研
- 多模块影响分析
- 并行信息收集

会非常有帮助。

### 15.3 Focus Chain

`Focus Chain` 用于在长任务中维持上下文焦点与任务推进节奏，适合持续数轮以上的实施过程。

### 15.4 Background Edit

启用后，`Vector` 可以在不频繁抢占编辑器焦点的前提下进行文件修改，更适合边看边协作的工作流。

### 15.5 Checkpoints

`Checkpoints` 可以在关键阶段保存进度，便于对比与回退。对于尝试性改动、分支方案验证和长链路任务尤其有价值。

### 15.6 Web Tools

在支持条件满足时，`Vector` 可以启用更强的 Web 浏览与搜索能力，用于补充在线信息获取与网页处理能力。

### 15.7 Jupyter Notebook 支持

在 Jupyter 场景下，`Vector` 还支持：

- 生成单元
- 解释单元
- 优化单元

适合数据分析、研究型工作流与实验性原型任务。

### 15.8 Auto Approve 与实验能力

在 `Features` 中，你还可以看到如自动批准、并行工具调用、原生工具调用、双重完成校验等高级选项。

需要特别注意的是，越高自动化的能力，越应该结合项目风险、团队规范与执行边界谨慎使用。

## 16. 常用命令与快捷操作

### 16.1 顶部入口命令

侧边栏顶部最常用的操作包括：

- `New Task`
- `MCP Servers`
- `History`
- `Account`
- `Settings`

### 16.2 编辑器与终端快捷入口

- 选中代码后可使用 `Add to Vector`
- 终端中可使用 `Add to Vector`
- 可使用 `Explain with Vector`
- 可使用 `Improve with Vector`
- 在 Source Control 中可使用 `Generate Commit Message with Vector`

### 16.3 键盘快捷方式

在默认配置下：

- 当编辑器中存在选区时，`Ctrl+'` 可将选中内容发送给 `Vector`
- 当编辑器中没有选区时，`Ctrl+'` 可将焦点跳转到 `Vector` 的聊天输入框

### 16.4 常用 Slash Commands

你可以在聊天输入框中使用下列命令：

- `/newtask`：基于当前上下文创建新任务
- `/deep-planning`：先做深入规划再进入实施
- `/smol`：压缩当前上下文
- `/newrule`：根据对话生成新的规则
- `/reportbug`：发起问题报告
- `/explain-changes`：解释 Git 变更、提交或分支差异

## 17. 推荐 Prompt 写法

`Vector` 的效果很大程度取决于你的任务描述方式。推荐采用“目标 + 范围 + 约束 + 参考”的写法。

### 17.1 功能开发

```text
请在当前项目中新增用户资料编辑能力。
要求：
1. 复用现有表单组件风格
2. 接口沿用当前用户模块的调用方式
3. 补充基础校验与测试
4. 先阅读 @/src/components/ 和 @/src/api/user.ts 再开始
```

### 17.2 Bug 修复

```text
请帮我定位并修复登录失败问题。
上下文：
1. 错误信息见 @terminal
2. 当前工作区报错见 @problems
3. 重点检查 @/src/pages/Login.tsx 和 @/src/api/auth.ts
```

### 17.3 代码理解

```text
请阅读 @/src/core/ 和 @/src/shared/，解释这个项目的核心数据流，并指出后续适合插入新能力的位置。
```

### 17.4 审查与重构

```text
请审查 @git-changes，重点关注潜在回归、边界条件和缺失测试。不要直接修改代码，先给出评审意见。
```

## 18. 最佳实践

### 18.1 先给目标，再给约束

不要只说“帮我改一下”。更好的方式是：

- 说明你要达成什么结果
- 说明不能破坏什么
- 指定参考实现
- 告诉它先分析还是直接动手

### 18.2 复杂任务优先使用 Plan

当任务满足以下任意条件时，建议先走 `Plan Mode`：

- 影响多个模块
- 涉及架构取舍
- 你自己也不确定实现路径
- 需要较高可控性

### 18.3 对关键路径给足上下文

高质量上下文通常比更长的自然语言描述更有效。优先提供：

- 关键文件
- 相关目录
- 报错信息
- Git 变更
- 外部文档链接

### 18.4 把审批当作质量关口

批准不是流程负担，而是质量控制点。建议在批准前重点确认：

- 是否改对了文件
- 是否符合项目惯例
- 是否引入多余改动
- 是否需要进一步测试

### 18.5 大任务拆小

即使 `Vector` 能处理复杂任务，也建议把目标拆成阶段：

1. 先理解现状
2. 再确认方案
3. 再实施第一阶段
4. 再检查结果并继续

这种方式通常更稳、更省上下文。

## 19. 常见问题

### 19.1 为什么 Vector 只分析、不改文件？

常见原因包括：

- 你当前处于 `Plan Mode`
- 你明确要求它先评估方案
- 它正在等待你的批准
- 当前任务信息不足，它还在补充上下文

如果你希望它开始执行，请确认：

- 已切换到 `Act Mode`
- 已明确说明可以开始落地
- 已对提议的变更进行批准

### 19.2 为什么它总是在读文件和搜索？

这是正常行为。`Vector` 的优势之一就是先理解项目，再执行操作。对于真实代码库，这种前置探索能明显降低误改概率。

### 19.3 为什么浏览器能力没有工作？

请依次检查：

- `Settings > Browser` 中是否启用了浏览器工具
- 本地页面是否已启动
- 远程浏览器模式是否配置正确
- 是否需要指定 Chrome 路径或启动参数

### 19.4 为什么终端命令没有正常执行？

请重点检查：

- 默认终端 Profile 是否正确
- Shell Integration 超时时间是否过短
- 当前终端执行模式是否适合你的环境
- 是否需要关闭激进终端复用以避免任务锁定问题

### 19.5 应该什么时候用 `/deep-planning`？

当任务很大、很复杂、涉及多个目录或需要明确实施清单时，优先使用 `/deep-planning`。小型改动不一定需要它。

### 19.6 使用自有 API Key 时要注意什么？

建议关注以下事项：

- 选择合适的 Provider 与模型
- 确认配额、速率限制和费用策略
- 按项目实际需求选择上下文长度与模型能力
- 对高成本模型，优先用于规划与关键问题阶段

## 20. 团队使用建议

如果你准备在团队中推广 `Vector`，建议采用以下方式：

- 统一推荐 Prompt 书写规范
- 为常见任务沉淀模板
- 将高频外部能力封装为 MCP
- 对高风险仓库优先要求 `Plan -> Act`
- 对自动批准、浏览器、终端执行设定明确边界

如果团队已经形成稳定流程，`Vector` 会更像一个可训练、可约束、可持续协作的工程代理，而不是一次性聊天工具。

## 21. 结语

`Vector` 适合的不是“只问一个问题就结束”的场景，而是“围绕一个真实工程目标持续推进”的工作方式。

当你给它明确目标、充足上下文和合理边界时，它可以成为 VS Code 中非常高效的智能研发助手：先理解，再规划，再执行，并始终把最终控制权保留给你。

如果你是第一次使用，建议从下面这条简单任务开始：

```text
请先阅读当前项目结构，告诉我最适合开始重构的一个模块，并说明理由。先不要改代码。
```

完成第一次分析后，再逐步尝试：

- 增量功能开发
- Bug 修复
- 测试补全
- 页面自动验证
- MCP 工具接入

这会是理解并用好 `Vector` 的最佳起点。
